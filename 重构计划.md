## 一、按优先级排列的重构计划

### Phase 0 — 安全与错误处理（全局修复，先治标）

> 跨文件、低风险、高收益的修复

#### P0.1 修复空异常捕获（全局）

| 来源 | 问题 | 文件 | 数量 |
|------|------|------|------|
| report + ai-review | 空 `except: pass` 静默吞掉异常 | `app.py` | 93 处 |
| report + ai-review | 空 `catch {}` 静默吞掉异常 | `static/scripts/app.js` | 25 处 |
| ai-review | 39.3% 的 except 被忽略 | `ai_commands.py` | 多处 |

**操作：**
- 所有 `except: pass` → 改为 `except Exception as e: logger.exception("...")`
- 所有 `catch {}` → 改为 `catch(e) { console.error("...", e); }` 或调用 `showError()`
- 在 `app.py` 中为 `_feed_stream_commands`、`_execute_commands_with_alias` 等模块添加异常日志

**优先级：** ⚡ 最高（风险最低，影响面最大）

#### P0.2 用户输入安全

| 来源 | 问题 | 操作 |
|------|------|------|
| ai-review | `handleKeyDown` 可能使用未转义的剪贴板内容 | 使用 `textContent` 替代 `innerHTML`，或加 DOMPurify |
| ai-review | `_locked_read_json` 路径遍历风险 | 使用 `os.path.basename()` 过滤文件名 |

**优先级：** ⚡ 高

### Phase 1 — `app.py`（糟糕指数 54.6）

> 核心问题：文件过长（2039 行）、`call_llm_stream` 上帝模块、复杂度峰值 _parse_commands_payload(37)、_execute_commands_with_alias(28)

#### 1.1 `_parse_commands_payload()` — 复杂度 37

| 来源 | 详情 |
|------|------|
| report L484 | 循环复杂度 37，为全项目最高 |
| ai-review | 大量重复逻辑模式，命令解析与别名展开耦合 |

**重构方案：**
1. 将**传统文本格式**和**JSON 格式**的解析拆为两个独立模块：`_parse_text_command(line)`、`_parse_json_command(line)`
2. 提取别名解析逻辑到 `_resolve_aliases(raw_params, alias_map)`
3. 使用**字典派发表**替代 if-else 链：
   ```python
   PARSERS = {
       "text": _parse_text_command,
       "json": _parse_json_command,
   }
   ```
4. 每个子模块复杂度控制在 ≤8

#### 1.2 `_execute_commands_with_alias()` — 复杂度 28

| 来源 | 详情 |
|------|------|
| report L631 | 循环复杂度 28 |
| ai-review | 别名展开与命令执行耦合 |

**重构方案：**
1. 将**命令类型分发**提取为策略模式：`_dispatch_command(cmd, params)`
2. 将**结果汇总逻辑**提取为 `_collect_results(results)`
3. 合并 `_execute_commands_with_alias` 与 `execute_circuit_command` 中重复的校验逻辑

#### 1.3 `call_llm_stream()` — 复杂度 67，模块过长（385+ 行）

| 来源 | 详情 |
|------|------|
| ai-review | 模块过长且复杂度最高 67，既是流处理器又是命令调度器 |
| report | `ai_generate_comments()` 复杂度 13，`execute_circuit_command()` 复杂度 14 |

**重构方案：**
拆分为 5-8 个内部模块，按职责：

```
call_llm_stream
├── _generate_system_prompt()        # 构建系统提示词（从 _build_autonomous_system_prompt 提取）
├── _stream_llm_response()           # 纯流式响应读取
├── _extract_sections_from_response()# 从完整回复提取 think/plan/build/observe/sum
├── _execute_build_commands()        # 执行 build 命令（已部分实现 _execute_batch_and_yield）
├── _execute_observe_verify()        # 执行验证用例
├── _persist_plan_and_summary()      # 保存 plan.md/summary.md
├── _check_done_condition()          # 判断是否完成
└── _prevent_loops()                 # 死循环/无进展检测
```

#### 1.4 `_format_feedback_text()` — 复杂度 50

| 来源 | 详情 |
|------|------|
| ai-review | 大量 if/elif 判断不同模板，建议策略模式或模板注册表 |

**重构方案：**
```python
FEEDBACK_TEMPLATES = {
    "build_success": "已执行 {n} 条命令，成功 {m} 条",
    "build_fail": "执行命令时出错：{error}",
    "verify_pass": "验证通过：{n}/{total} 用例",
    "verify_fail": "验证失败：{expected} != {actual}",
    "no_progress": "连续 {n} 轮无进展，停止执行",
    ...
}

def _format_feedback_text(feedback):
    template = FEEDBACK_TEMPLATES.get(feedback["type"])
    return template.format(**feedback) if template else ""
```

| 来源 | 详情 |
|------|------|
| report L223 | 循环复杂度 33 |

**重构方案：**

```javascript
const KEY_ACTIONS = {
    '1': () => setTool('select'),
    '2': () => setTool('input-toggle'),
    '3': () => addElement('AND'),
    '4': () => addElement('OR'),

    'Delete': () => deleteSelected(),

function handleKeyDown(e) {
    const action = KEY_ACTIONS[e.key];
    if (action) { action(); e.preventDefault(); }
}

|------|------|

                return json.load(f)
        except Exception:
            return None
```
